# Personal PRD Profile

## Current Stage

The user's PRD capability has moved from feature clarification into system-building. Their PRDs now frequently involve AI content generation, AI image workflows, prompt/configuration platforms, book-linking/pinshu flows, data dashboards, revenue tracking, and operational controls.

## Observed Evolution

| Stage | Signal | Typical PRDs | Growth |
|---|---|---|---|
| Link and tool repair | Fix broken product paths or missing internal tools | Pinshu tagging/jump links, AI image review backend | Can isolate concrete operational pain and define P0 implementation needs |
| Data observability | Build dashboards and events | AI image events, pinshu revenue metrics, token cost boards | Starts connecting features to measurable business effects |
| Experiment strategy | Validate AI/image/pinshu directions | AI image experiment phase 2, ad strategy changes | Starts designing larger business experiments instead of shipping one function |
| Platform configuration | Make repeated work configurable | Pinshu backend config, image ad slots, one-click on/off, prompt tuning | Moves from one-off delivery into operational systems |

## Current Strengths

- Converts vague AI/content opportunities into concrete product surfaces.
- Thinks in backend tools, configuration, batch operations, and data visibility.
- Writes enough detail for engineering to start implementation.
- Increasingly recognizes revenue, IPU, Token cost, exposure, and funnel metrics.

## Repeated Improvement Areas

- Put the business hypothesis before the feature list.
- Explain why each metric is sufficient to decide success or failure.
- Add a necessity/workload gate before adding gray release, experiment, training, or heavy review work. Not every requirement needs all ceremonies.
- When experiments are necessary, structure them as hypothesis, variable, sample, grouping, threshold, duration, and next action; when they are unnecessary, state why online observation is enough.
- Add state/permission/failure/rollback/history requirements for backend tools.
- Write developer-facing requirements at page level: display elements, interaction method, strategy/rule, fallback logic, and acceptance criteria.
- Define post-launch review timing and decision rules.

## Learning PRD Direction

The learning PRD currently shows a useful target style: stronger business context, cooperation background, target metrics, competitor/product reference, and broader launch context before entering functional details.

When comparing future learning PRDs, extract what they do better by type:

- Business narrative
- Metric threshold
- User journey
- Competitive reference
- Experiment/rollout plan
- Necessity/workload tradeoff
- Developer-facing clarity
- Cross-team coordination
- Risk control
